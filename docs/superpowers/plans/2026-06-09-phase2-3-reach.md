# Phase 2+3: Reach & Visibility — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add notifications (Discord/Telegram + heartbeat), ICS export, a `serve` command (hosted ICS feed + status page = Phase 3's Option-C layer), Docker support, and release 0.3.0.

**Architecture:** Notifications are a side channel: a `Notifier` protocol fed by the CLI after runs and on errors; channel secrets live in env vars. The state file becomes the source of truth for "what is on the calendar" — `TrackedEvent` is enriched with display fields so the ICS file can be regenerated wholesale from state each run (no second reconcile sink). `serve` wraps the existing pipeline in a background loop thread plus a stdlib HTTP server.

**Tech Stack:** Existing stack only; HTTP server via `http.server` (stdlib, no new deps).

**Design decisions locked here:**
- `TrackedEvent` gains `summary: str = ""` and `start: str = ""` (ISO). v1 state files load fine via defaults; events lacking display data are skipped by ICS until they regenerate. `State` gains `last_heartbeat: str | None = None`. `STATE_VERSION = 2`.
- `[notify]` config: `on_events`, `on_errors` (bools, default true), `heartbeat_hours` (int, default 0 = off). Channels activate from env: `DISCORD_WEBHOOK_URL`, `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID`. Notification failures are logged, never crash a run.
- `[ics]`: `enabled` (default false), `path` (default `ftmo-events.ics`). Regenerated after every non-dry run. UID = `<event_key>@ftmo-calendar`; times in UTC (`Z`); `VALARM` per configured reminder minutes.
- `[serve]`: `host` (default `0.0.0.0`), `port` (8080), `sync_interval_minutes` (360). `ftmo-calendar serve` = periodic sync loop in a daemon thread + HTTP endpoints `GET /feed.ics` (ICS, forced enabled), `GET /status` (HTML), `GET /healthz` (JSON: ok, last run, last error). Sync failures in serve mode are recorded + notified but the server stays up.
- Phase 3 "more sources" stays demand-driven — explicitly NOT built.
