# Phase 3 Deep: Feed-First Hosting — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the hosted-feed path truly zero-Google (one LLM key = a feed a whole group can use), make first contact frictionless (dry-run with no setup), and harden serve-mode operations. Release 0.4.0.

**Deep-review findings driving this plan:**
1. `serve`/`run` construct Google credentials unconditionally — feed-only hosting is impossible without Google setup (architectural gap vs the Phase 3 vision).
2. `--dry-run` builds the calendar sink it never uses (reconcile short-circuits before sink calls in dry-run) — new users hit Google errors on their first preview command.
3. `run_sync_loop` re-notifies the same persistent error every interval — 4 identical pings a day for one expired token.
4. ICS lacks `REFRESH-INTERVAL`/`X-PUBLISHED-TTL` and `DESCRIPTION`; `/healthz` lacks `next_run`; status page lacks "next event"; Dockerfile lacks `HEALTHCHECK`; cli module docstring omits `serve`.

### Task 1: Feed-only mode + sink factoring

**Files:** `config.py` (`CalendarConfig.enabled: bool = True`), new `sinks/null.py` (`StateOnlySink`: find→None, create→`"ics:<key>"`, delete→no-op), `cli.py` (`_build_sink(config, dry_run)` returning StateOnlySink when dry-run or calendar disabled — credentials only touched for a real Google run; `_cmd_serve`/`_cmd_run` force `ics.enabled=True` when calendar disabled, else a run produces nothing; `auth` explains itself when calendar disabled); tests in `tests/test_null_sink.py`, `tests/test_cli.py`, `tests/test_config.py`.
- [ ] Tests: StateOnlySink contract; `_build_sink` returns StateOnlySink for dry_run without touching auth (config pointing at empty dir must not raise); calendar disabled → StateOnlySink + ics forced on; config default enabled=true.
- [ ] Gates green; commit.

### Task 2: ICS feed quality

**Files:** `sinks/ics.py`, call sites in `cli.py`; tests in `tests/test_ics.py`.
- [ ] `render_ics(state, reminders, *, source_url="", refresh_minutes=0, now=None)`: emit `REFRESH-INTERVAL;VALUE=DURATION:PT<m>M` + `X-PUBLISHED-TTL:PT<m>M` when refresh_minutes>0; per-event `DESCRIPTION:Source: <url>\nCreated by AutoFtmoCalendar` when source_url given.
- [ ] Call sites pass `source_url=config.source.url`, `refresh_minutes=config.serve.sync_interval_minutes`.
- [ ] Tests for both; gates green; commit.

### Task 3: Serve operations hardening

**Files:** `server.py`, `tests/test_server.py`.
- [ ] `run_sync_loop`: notify `on_error` only when `str(error)` differs from the previously notified error; reset on success. Tests: same error twice → one notification; A,B → two; A,success,A → two.
- [ ] `ServerStatus(interval_seconds=...)`: `snapshot()` adds `next_run` (last_run + interval, ISO) and `ok` stays None-error-based. `/healthz` therefore exposes overdue-detection. Test.
- [ ] Status page: "Next event:" line — earliest tracked event with `start > now` (from state). Test.
- [ ] Gates green; commit.
